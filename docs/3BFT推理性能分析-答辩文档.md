# 3B Full Fine-tuning 模型推理性能分析

## B737 驾驶舱 AI 副驾驶系统 — 答辩技术文档

---

## 一、概述

本系统采用 Qwen2.5-3B-Instruct 进行全参数微调（Full Fine-tuning），在 B737-800 驾驶舱 AI 副驾驶场景中实现了**毫秒级推理延迟**。模型从加载到首 token 生成仅需 2.3 秒，单步 decode 约 15ms，整体推理性能远超同场景下 QLoRA 微调的 7B 模型。

本文档从模型规模、训练方式、推理架构、硬件适配四个维度，系统分析 3B Full FT 实现快速推理的根本原因。

## 三、核心原因分析

### 3.1 模型规模效应：参数减半，计算量线性下降

Transformer 模型单次前向传播的计算量主要由注意力机制和前馈网络（FFN）构成。对于自回归解码阶段（单 token 输入），计算量为：

$$F_{\text{decode}} \approx 2 \cdot N_{\text{params}} + 4 \cdot n_{\text{layers}} \cdot d_{\text{model}} \cdot n_{\text{kv}}$$

其中 $N_{\text{params}}$ 为总参数量，$n_{\text{kv}}$ 为 KV-Cache 长度。

由于 3.1B 参数约为 7.6B 的 40%，每步 decode 的计算量直接下降到约 **40%**。这是推理加速的最基础物理原因——**在相同硬件上，更少的浮点运算必然意味着更低的延迟**。

此外，更小的模型意味着：
- 更少的显存带宽压力（权重矩阵更小，每次从 HBM 读取的数据量更少）
- 更少的注意力头数，多头注意力的并行开销更低
- 更少的 Transformer 层数，逐层串行传播的总延迟更低

---

### 3.2 Full Fine-tuning 的架构优势：原生权重，零额外开销

这是本系统推理速度的关键结构优势。需要理解 QLoRA 推理的两种模式及其开销：

#### 3.2.1 QLoRA 在线合并模式

QLoRA 在推理时，每个线性层的计算为：

$$y = W_{\text{base}} \cdot x + \alpha \cdot (A \cdot B) \cdot x$$

其中 $W_{\text{base}}$ 为冻结的 NF4 量化基座权重，$A \in \mathbb{R}^{d \times r}$ 和 $B \in \mathbb{R}^{r \times d}$ 为 LoRA 适配器矩阵。这引入了以下开销：

1. **额外矩阵乘法**：每个带有 LoRA adapter 的层需要额外计算 $A \cdot B \cdot x$，在 attention 的 Q/K/V/O 投影和 FFN 的中间层均有此开销
2. **NF4 反量化**：`bitsandbytes` 的 NF4 格式在每次 forward 时需要将 4-bit 量化值还原为 bf16，增加反量化 kernel 的启动开销
3. **内存碎片化**：base 权重和 adapter 权重存储在不同内存区域，访存模式不连续，Cache 命中率降低

#### 3.2.2 QLoRA 离线合并模式

可以将 LoRA adapter 提前合并进 base 模型：

$$W_{\text{merged}} = W_{\text{base}} + \alpha \cdot A \cdot B$$

但这需要额外的预处理步骤，且合并后的模型失去了模块化部署的优势（无法热切换 adapter）。

#### 3.2.3 Full FT 的纯前向路径

相比之下，Full Fine-tuning 的权重本身就是原生 bf16 矩阵，前向传播是单次矩阵乘法：

$$y = W_{\text{fine-tuned}} \cdot x$$

**没有任何 adapter 合并、反量化、或分支计算**。计算图是纯粹的单路径，编译器（torch.compile / CUDA Graph）有更大的优化空间，GPU 的 Tensor Core 利用率接近理论峰值。

---

### 3.3 单卡满载部署：零通信开销

3B Full FT 在 bf16 精度下 VRAM 占用仅约 **3.8 GB**，完全容纳于一张 RTX 3090（24 GB）内。`device_map="auto"` 自动将全部分片放在一张卡上。

对比之下，7B QLoRA 模型（即使基座为 NF4）加上 adapter、KV-Cache、激活值内存后，可能突破单卡 24 GB 上限，触发以下退化：

| 部署模式 | 通信开销 | 延迟影响 |
|---|---|---|
| 单卡全载（3B FT） | 0 | 基准延迟 |
| 跨卡张量并行（如 7B） | PCIe/NVLink 传输激活值 | +30-50% |
| CPU Offload（如 ZeRO-3） | PCIe 带宽瓶颈（~16 GB/s） | +5-10× |

对于本系统的实时推理场景（需要毫秒级响应），**单卡部署是延迟的硬约束**。3B 模型恰好在这个约束下达到了最优解——足够小以单卡全载，又足够大以捕获飞行领域的专业判断能力。

---

### 3.4 KV-Cache 流式推理：Prefill 一次，Decode 极简

本推理服务采用**手动 KV-Cache 管理**的 token-by-token 流式生成机制，而非调用 HuggingFace `model.generate()`。

#### 3.4.1 两阶段流程

```
┌─────────────────────────────────────────────────────────┐
│  Prefill 阶段（仅一次）                                    │
│  ┌──────────────────────────────────────────────────┐   │
│  │ prompt(~500 tokens) → Model Forward → KV Cache   │   │
│  │ 计算量: O(n_prompt² · d_model)                    │   │
│  │ 延迟: ~200-500ms（一次性开销）                      │   │
│  └──────────────────────────────────────────────────┘   │
│                          ↓                               │
│  Decode 阶段（每 token 一次，最多 64 步）                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ 1 token + KV Cache → Model Forward → 1 token     │   │
│  │ 计算量: O(n_kv · d_model)                         │   │
│  │ 延迟: ~15ms / step                                │   │
│  └──────────────────────────────────────────────────┘   │
│                          ↓                               │
│  EOS 终止（实际 10-30 tokens）                            │
└─────────────────────────────────────────────────────────┘
```

#### 3.4.2 与 `model.generate()` 的对比

HuggingFace 的 `generate()` 虽然也使用 KV-Cache，但内部有额外的逻辑开销（logits processor 遍历、beam search 状态管理、停止条件轮询）。本系统直接操控 `past_key_values`，**去除了所有非必要的 Python 层开销**，并在每步生成后通过 `await websocket.send_text()` 让出事件循环，保证服务器对心跳和并发连接的响应性。

---

### 3.5 bf16 + Ampere Tensor Core：精度与速度的最佳平衡点

#### 3.5.1 精度选择分析

| 精度 | 内存占用（3B） | RTX 3090 Tensor Core 支持 | 数值稳定性 |
|---|---|---|---|
| fp32 | ~12.4 GB | ✓（慢） | 最优 |
| fp16 | ~6.2 GB | ✓（快） | 需 loss scaling |
| bf16 | **~6.2 GB** | ✓（快，原生） | 与 fp32 同动态范围 |
| int8 | ~3.1 GB | ✗（需反量化） | 精度损失 |
| NF4 | ~1.6 GB | ✗（需反量化） | 精度损失 |

bf16 的关键特性：**指数位与 fp32 相同（8 位），尾数位截断为 7 位**。这意味着 bf16 的动态范围可达 $10^{38}$，不会像 fp16 在 $2^{16} \approx 65504$ 处溢出。对于 Transformer 中常见的 softmax 和 LayerNorm 操作，bf16 无需梯度缩放即可直接计算。

RTX 3090 基于 Ampere 架构（SM 8.6），其第三代 Tensor Core **原生支持 bf16 输入**，单 SM 的 bf16 TFLOPS 可达 fp32 的 4 倍。这保证了 3B 模型在 bf16 下的推理性能基本达到硬件上限。

#### 3.5.2 量化推理的性能陷阱

值得注意的是，虽然 int8/NF4 量化能进一步降低显存占用，但**并不一定加速推理**：

1. 量化 kernel 的 GPU 利用率通常低于原生 bf16 matmul
2. 反量化操作引入额外的寄存器压力和指令发射开销
3. 对于小 batch size 的推理（batch=1），内存带宽不是瓶颈，计算吞吐才是

对于 3.8 GB 的小模型，bf16 在延迟和吞吐之间取得了最优平衡。

---

### 3.6 短输出与提前终止：任务特性的天然加速

驾驶舱 AI 副驾驶的输出规范要求**不超过 50 字**，且系统设定 `max_tokens=64`。实际运行中：

- 正常报告（如"正常，各参数均在标准范围内"）：**8-15 tokens**
- 告警响应（如"发动机火警！立即执行发动机火警检查单"）：**12-20 tokens**
- 复杂态势说明：**20-30 tokens**

对比通用聊天场景动辄 512-2048 tokens 的输出，本系统的 decode 步数减少了 **1-2 个数量级**。每次遇到 EOS token 即终止，不存在冗余生成。

这是一个**任务定义层面**的加速——不是优化出来的，而是设计出来的。

## 六、答辩要点提示

1. **"为什么不用更大的模型？"**
   - 3B Full FT 的 eval_loss (0.0873) 已经显著优于 7B QLoRA (0.12)，精度和效率同时更优
   - 驾驶舱场景需要毫秒级实时响应，单卡部署是硬约束，3B 是当前硬件下的帕累托最优

2. **"Full FT 和 QLoRA 的 trade-off 是什么？"**
   - Full FT 推理路径更短、延迟更低，但训练代价更高（3B Full FT 训练 11.6 小时 vs 7B QLoRA 2.6 小时）
   - 对于推理密集型部署场景（推理量 >> 训练量），Full FT 是更优选择

3. **"如何进一步优化？"**
   - TensorRT-LLM / vLLM 集成，利用 PagedAttention 和 CUDA Graph
   - INT8 KV-Cache 量化，进一步降低显存、提高 batch size
   - torch.compile 或 `--torch-run` 编译静态图消除 Python overhead

4. **"这个方案的通用性如何？"**
   - 方法论通用：小模型 Full FT + 单卡 + KV-Cache 流式推理的策略适用于任何延迟敏感、输出短、领域固定的推理场景
   - 实际瓶颈在不同领域的数据质量和训练覆盖度，而非推理架构
