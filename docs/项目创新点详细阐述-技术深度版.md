## 一、空间哈希网格 — 23.5 万航路点 O(1) 实时邻近查询

### 做了什么

为导航显示器（ND）的地图叠加层设计了专用的 **Grid-based Spatial Hash**。

```
1° 纬度带 × 动态经度宽度（1.0/cos(lat)，约 60 NM/格）
64-bit 打包键：(lat_band << 32) | lon_sector
2,003 桶哈希表，~235k 条目全堆分配
航向优先 5×5 邻域搜索：按余弦相似度排序邻域，优先飞机前方网格
```

### 为什么不用现成的方案

| 方案 | 问题 |
|---|---|
| **线性扫描 235k 条目** | 每帧 O(n) → 60 FPS 下 CPU 被吃满 |
| **朴素 KD-Tree** | 动态插入开销大，纬度带下难以平衡 |
| **GeoHash / S2** | 外部库依赖，高纬度区域精度不均 |
| **四叉树** | 节点分配碎片化，航向感知需额外处理 |

### 为什么自研空间哈希

1. **查询模式特殊**：ND 只需矩形视口范围内的点，不是 k-NN。网格哈希天然支持 O(1) 邻域枚举。
2. **航向优先**：飞机前方 120° 范围内的导航点才是"有意义"的。我们设计了按航向余弦排序的邻域搜索——先搜正前方网格，再逐步外扩——确保前 80 个结果中包含所有屏幕可见点。
3. **动态纬度带**：赤道 1° ≈ 60 NM，但在北纬 60° 只有 30 NM。用 `1.0/cos(lat)` 动态调节经度带宽，保证所有格子的实际地理面积均匀。
4. **零依赖**：纯 C + `calloc`/`free`，不存在任何第三方库依赖，编译链极简。

### 性能数据

- 23.5 万条目加载：~2 秒（一次性）
- 单帧查询延迟：**< 1ms**（80-160 条目）
- 对比线性扫描：**~200× 加速**

## 二、A* 航路寻路引擎 + 链式前向星图

### 做了什么

从零实现了完整的**图论航路规划系统**——不是简单的航路点列表，而是可自动计算最优路径的图引擎。

### 架构

```
Nav 数据库（航路点 + 机场）
        ↓
链式前向星图构建（有向有权边，haversine 大圆距离权重）
        ↓
A* 搜索（haversine 启发函数，二叉最小堆开放集，世代标记闭合集）
        ↓
飞行计划（waypoint[] + 距离 + ETA + 燃油估算）
        ↓
LEGS 页展示 + ND 渲染
```

### 为什么用链式前向星

| 邻接表方案 | 每条边遍历 | 内存 | 插入 |
|---|---|---|---|
| `vector<vector<Edge>>` | O(1) | 每边 2 指针 | O(1) |
| 邻接矩阵 | O(V) | V² | — |
| **链式前向星** | **O(1)** | **每边 3 int** | **O(1)** |

链式前向星（Chain Forward-Star）是 ACM 竞赛级图存储结构：用 `head[u]` 索引 + `edge[].next` 链表实现 O(1) 边遍历，比 `vector<vector<Edge>>` 更紧凑（无 `std::vector` 的 capacity 预留和 realloc），比邻接矩阵节省 99.9% 空间。

### 为什么用 A* 而非 Dijkstra

```
Dijkstra：北京 → 上海，1,000 节点，展开 ~600 节点
A*（harversine 启发）：展开 ~80 节点 → 7.5× 加速
```

Haversine 大圆距离作为启发函数且**保证可纳**（地球表面最短路径不会短于大圆距离），因此 A* 解是数学上可证明的最优解。

## 三、X-Plane 12 双向原生 UDP 协议栈

### 做了什么

完整逆向并实现了 X-Plane 12 的原生 UDP 通信协议栈——不使用任何第三方插件（如 XPUIPC）。

### 协议矩阵

| 方向 | 协议 | 端口 | 格式 |
|---|---|---|---|
| XP12 → Cockpit | **RPOS** | 49005 | 5B 固定 |
| XP12 → Cockpit | **DSEL** × 27 | 49005 | 6B 固定，数据组选择 |
| XP12 → Cockpit | **DATA** | 49005 | N×36B，多组打包 |
| Cockpit → XP12 | **DREF** | 49000 | **509B 固定包**（XP12 新格式） |
| Cockpit → XP12 | **CMND** | 49000 | ~21B，FMC 键盘镜像 |
| Cockpit → XP12 | **RREF** | 49000 | 413B 固定，25 条告警 DREF 订阅 |

### 攻克的技术难点

1. **XP12 数据组索引重排**：XP11→XP12 升级中 160+ 数据组编号全部改变，手动逐组测绘。例如 "groundspeed" 在 XP11 是 data group 3，在 XP12 变为 group 20。

2. **DATA 包第 5 字节差异**：XP11 的 DATA 包标识字节是 `0x00`，XP12 改为 `0x2A`（`"DATA*"` 的五字节标记首字符）。不识别这个差异会导致所有 DATA 包被丢弃。

3. **DREF 509B 固定包协议**：XP12 要求所有 DREF 写入包统一为 509 字节——前 5 字节 `"DREF\0"`，中间 dataref 路径字符串，末尾 float 值 + padding。而 XP11 使用可变长 42-44 字节。这是 XP12 为防止缓冲区溢出新增的协议约束。

4. **UDP 粘包修复**：3 条 DREF 零间隔连续发送会导致 XP12 错误切分包边界 → `38 bytes vs 504 bytes` 报错。在每条 DREF 间插入 `SDL_Delay(15)` 消除粘包。

5. **RREF 订阅系统**：25 条告警状态 DREF 通过 RREF 批量订阅，XP12 以 1-4 Hz 频率主动推送 float 值——避免了轮询开销，将告警数据采集带宽从 ~24 KB/s 降至 ~300 B/s。

## 四、Qwen2.5-3B Full Fine-tuning — AI 副驾驶推理架构

### 模型选择：为什么从 ChatGLM3-6B QLoRA 切换到 Qwen2.5-3B Full FT

| 维度 | ChatGLM3-6B QLoRA | Qwen2.5-3B Full FT | 抉择理由 |
|---|---|---|---|
| 参数量 | 7.6B | 3.1B | 减少 60% FLOPS |
| 训练方式 | QLoRA 4-bit (r=16) | 全参数微调 bf16 | 推理无 Adapter 额外计算 |
| VRAM | 8-10 GB | **3.8 GB** | 单卡 3090 富裕容纳 |
| 推理延迟 | ~25-35ms/token | **~15ms/token** | 毫秒级实时响应 |
| eval_loss | ~0.12 | **0.0873** | 精度反超 27.3% |
| 训练时间 | 2.6h | 11.6h | 训练代价可接受（一次性） |

### Full Fine-tuning 推理优势

QLoRA 推理的每个线性层计算路径：

```
y = W_base · x + α · (A · B) · x
     ↑ 需 NF4 反量化         ↑ Adapter 额外矩阵乘法
```

Full FT 的纯前向路径：

```
y = W_fine-tuned · x
     ↑ 原生 bf16 矩阵，单次乘法，Tensor Core 直接加速
```

**无 Adapter 合并、无反量化、无分支计算**。计算图是纯粹的单路径——编译器（CUDA Graph）可深度优化，GPU Tensor Core 利用率接近理论峰值。

### 手动 KV-Cache 流式推理

不使用 HuggingFace `model.generate()`，而是手动管理 `past_key_values`：

```
Prefill（一次性）：prompt(~500 tokens) → Model Forward → KV Cache
                                       延迟: ~200-500ms

Decode（每 token）：1 token + KV Cache → Model Forward → 1 token
                                       延迟: ~15ms / step

实际输出：8-30 tokens（飞行建议 ≤50 字），总延迟 ~350-800ms
```

去除 HF generate 的 logits processor 遍历、beam search 状态管理等 Python 层开销，每步生成后通过 `await websocket.send_text()` 让出事件循环，保证心跳和并发连接的响应性。

### 自研 WebSocket RFC 6455 C 客户端

从零手写，仅依赖 Winsock2，零第三方库：

- **937 行 C 代码**
- 完整的 WebSocket 握手（Base64 编码 `Sec-WebSocket-Key` + SHA1 + HTTP Upgrade）
- 帧封装（FIN + opcode + mask + payload）、帧解析、Ping/Pong
- 指数退避重连（2s → 30s）
- 事件驱动状态推送（参数变化超阈值才发送，≤200ms 间隔，离散事件立即推送）
- 推理门控（服务器忙时不推送新状态，离散事件排队等待）
- 流式 token 逐字接收 + 帧重组 → 渐进式 UI 显示

### 训练数据：39 类告警 × 80 飞行剖面

全自动数据生成管道：

```
Python 飞行剖面模拟器（替代失真严重的 C mock_data.c）
  → 62,000 帧剖面池 + 参数扰动 + 6 种中文文本变体
  → 37,700 条 ShareGPT 格式训练样本
  → 23 条规则模板（13 GPWS + 10 系统参数，阈值精确提取自 alert_system.c）
```

## 五、C 层 RREF 告警状态读取

### 架构决策：告警检测与 AI 建议分离

AI 模型不是告警检测器——告警由 C 层硬信号完成，AI 只做态势分析。

```
X-Plane 12 (port 49000)
    ← RREF request (413B): 订阅 25 条告警状态 DREF @ 1-4 Hz
    → RREF response (12B):  4B "RREF" + 4B request_id + 4B float value
        ↓
xplane_parse_rref() → FlightDataValues.dref_* (21 bool + 4 float)
        ↓
alert_system_get_active_alerts() → int[39] 输出流
        ↓
int[39] → AI 模型 input（作为告警上下文注入 prompt）
```

**为什么这样设计**：AI 模型的参数量决定了它在判断告警阈值方面的精度不如硬编码规则——"发动机 EGT > 950°C"这样的确定性判断不需要模型。模型的价值在于**组合推理**：当多个告警同时触发时，综合分析态势，给出优先级建议。

## 七、哈希表 + 二分搜索 + AVL 树的三层数据查找架构

### 查什么 vs 用什么

| 查找场景 | 数据结构 | 复杂度 | 数据量 | 为什么不用 AVL |
|---|---|---|---|---|
| 航路点 ident → 坐标 | **哈希表** | **O(1)** | 235k | AVL 需 ~18 次比较，哈希 1 次 |
| 机场 ICAO → Airport | **二分搜索** | O(log n) | 179 | 连续数组，CPU 缓存友好 |
| 空间邻近查询 | **空间哈希** | O(1) 近似 | 235k | AVL/KD-Tree 无航向感知 |
| LEGS 字母排序 | **AVL 树** | O(log n) | ≤128 | 排序是 AVL 天然优势 |

### 为什么机场用二分搜索而不用哈希表

179 个机场数量太少，哈希表的内存开销（桶数组 + 链表节点）超过二分搜索的零额外开销。同时排序后连续数组的 bsearch 完美利用 CPU L1/L2 缓存行预取，实际速度可能反超哈希表。

### 为什么航路点用哈希表而不用二分搜索

23.5 万条目上二分搜索需 ~18 次比较，每次比较可能触发缓存缺失。哈希表用 **500,009 个桶**（大质数，负载因子 0.47），一次 hash + 平均不到一次链表遍历即可命中。

## 十一、客舱 3D 地球航线显示 — OpenGL 真实地球纹理投影

### 做了什么

使用 OpenGL 构建了独立的 3D 地球客舱航线显示器，**将高清地球卫星照片纹理投影到 UV 球体上**，实时显示飞行轨迹：

```
assets/earth_daymap.jpg（等距矩形投影高清地球纹理）
        ↓ SDL_image 加载 → GL 纹理（MIP-map 线性过滤）
        ↓ 映射到 UV 球体网格（64 纬度 × 128 经度 = 8,192 三角面）
        ↓ OpenGL Display List 预编译球体网格（零帧间重建开销）
        ↓ 3D 渲染：飞行路线 + GPS 航迹 + 航路点标记 + 飞机图标
        ↓ 2D 正交投影叠加：HUD 数据栏 + 进度条
```

### 核心技术点

**1. 等距矩形投影 → UV 球体**

真实地球卫星照片使用等距矩形投影（equirectangular projection），纹理坐标 `(u,v)` 直接对应经纬度 `(lon/360, lat/180)`。UV 球体的每个顶点按球面坐标计算法线，配合 `GL_REPEAT` 纹理包裹实现无缝地球。

**2. 球体网格预编译**

64×128 段 UV 球体（8,192 个三角面）在初始化时编译为 OpenGL Display List（`glGenLists` + `glCallList`），后续每帧只需一次 `glCallList` 调用，**零 CPU 端顶点计算开销**。

**3. 3D 空间中的航线叠加**

- 飞行计划航线：青色连线，在球面上方 `ROUTE_OFFSET = 0.004` 处渲染，避免 Z-fighting
- GPS 航迹（已飞路径）：黄色实线，2048 点环形缓冲
- 航路点标记：菱形/方形彩色标记，`MARKER_OFFSET = 0.008` 处悬停
- 飞机图标：`plane.png` 精灵纹理，在当前位置按航向旋转，ease-out 平滑插值

**4. 平滑相机轨道动画**

相机距离从默认 1.15（近地轨道视角）到 0.6-3.0 可调，22° 倾斜角。Y 轴旋转随飞机航向平滑过渡（`target_rot_y` 插值），保持飞机大致面向前方。

**5. 2D 正交投影 HUD 叠加**

3D 渲染完成后切换为正交投影，绘制：
- 顶部标题栏：出发地 → 目的地 + 天气简况
- 底部数据栏：GS / ALT / HDG / OAT / 距目的地 / ETA
- 航程进度条（3px 高，洋红色填充）
- 所有文字通过 `TTF_RenderUTF8_Blended` → GL 纹理 → 四边形渲染

### 为什么不用高德/Google Maps API

| 方案 | 问题 |
|---|---|
| Web JS API（高德/Google） | 需浏览器/WebView，无法嵌入 C + OpenGL 渲染管线；需网络 |
| 离线瓦片地图 | 需数千张预切图片，存储和加载复杂 |
| **等距矩形纹理 + UV 球体** | **单张 JPG（几 MB），GL 直接采样，零网络依赖，无限缩放** |

### 为什么这个方案更好

1. **真正的 3D 球体**：航线跟随球面曲率，不是平面地图上的直线——大圆航线在球面上自然呈现为弧线
2. **零网络依赖**：单张纹理文件本地加载，客舱显示器不需要互联网连接
3. **完全可控的渲染**：OpenGL 原生渲染，不依赖浏览器引擎的合成开销
4. **与驾驶舱渲染管线统一**：同进程内 SDL + OpenGL，共享 FlightData 内存快照，零序列化开销
