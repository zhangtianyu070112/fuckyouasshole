## 三、通信协议

### 3.1 连接参数

| 项目 | 值 |
|------|-----|
| 协议 | WebSocket (RFC 6455) |
| 端口 | 8090 |
| 路径 | `/api/copilot/stream` |
| 编码 | JSON 文本帧, UTF-8 |
| 心跳 | WebSocket Ping, 30s 间隔 |
| 连接模式 | 持久连接（不断开，复用） |
| 最大连接数 | 1（驾驶舱单客户端） |

### 3.2 开发机 → 服务器：飞行状态推送

**推送时机**：事件驱动，状态变化超过阈值时推送，最小间隔 200ms。

#### JSON 报文格式

```json
{
  "alt_ft": 35000,
  "agl_ft": 34000,
  "ias_kts": 280,
  "vs_fpm": 50,
  "mach": 0.785,
  "roll_deg": 2.1,
  "pitch_deg": 2.5,
  "heading": 125,
  "n1_pct": [60.2, 60.8],
  "egt_c": [700, 705],
  "flap_ratio": 0.0,
  "gear_down": 1,
  "fuel_lbs": 8500,
  "oat_c": -39,
  "ap_engaged": 1,
  "master_warning": 0,
  "master_caution": 0
}
```

#### 字段定义

| 字段 | 类型 | 单位 | 说明 |
|------|------|------|------|
| `alt_ft` | float | ft | 气压高度 (MSL) |
| `agl_ft` | float | ft | 无线电高度 (AGL) |
| `ias_kts` | float | knots | 指示空速 |
| `vs_fpm` | float | ft/min | 垂直速度（正=上升） |
| `mach` | float | — | 马赫数 |
| `roll_deg` | float | degrees | 横滚角（左负右正） |
| `pitch_deg` | float | degrees | 俯仰角（抬头上仰为正） |
| `heading` | float | degrees | 磁航向 |
| `n1_pct` | [float, float] | % | 左/右发 N1 转速 |
| `egt_c` | [float, float] | °C | 左/右发排气温度 |
| `flap_ratio` | float | 0.0~1.0 | 襟翼放出比例（0=全收, 1=全放） |
| `gear_down` | int | 0/1 | 起落架：0=收上, 1=放下 |
| `fuel_lbs` | float | lbs | 剩余燃油总量 |
| `oat_c` | float | °C | 外界大气温度 |
| `ap_engaged` | int | 0/1 | 自动驾驶：0=断开, 1=接通 |
| `master_warning` | int | 0/1 | 主警告灯：0=灭, 1=亮 |
| `master_caution` | int | 0/1 | 主警戒灯：0=灭, 1=亮 |

#### C 端推送阈值（仅在此范围外变化才推送）

| 参数 | 阈值 | 说明 |
|------|------|------|
| `alt_ft` | ±50 ft | 高度变化 |
| `agl_ft` | ±50 ft | 无线电高度 |
| `ias_kts` | ±5 kts | 空速变化 |
| `vs_fpm` | ±100 fpm | 垂直速度变化 |
| `mach` | ±0.01 | 马赫数变化 |
| `heading` | ±3° | 航向变化 |
| `roll_deg` | ±2° | 横滚变化 |
| `pitch_deg` | ±1° | 俯仰变化 |
| `n1_pct` | ±2% | N1 变化 |
| `egt_c` | ±10°C | EGT 变化 |
| `flap_ratio` | ±5% (0.05) | 襟翼位置变化 |
| `fuel_lbs` | ±200 lbs | 燃油变化 |
| `gear_down` | 任意变化 | 离散量 |
| `ap_engaged` | 任意变化 | 离散量 |
| `master_warning` | 任意变化 | 离散量 |
| `master_caution` | 任意变化 | 离散量 |

### 3.3 服务器 → 开发机：Token 流式推送

每个 token 一个独立 WebSocket 文本帧，推理完成后发送结束标记。

#### 正常推送

```
{"token": "当"}
{"token": "前"}
{"token": "巡"}
{"token": "航"}
{"token": "状"}
{"token": "态"}
{"token": "，"}

...更多 token...

{"token": "。"}
{"done": true}
```

#### 错误推送

```
{"error": "invalid_json"}        ← JSON 格式错误
{"error": "inference_failed"}    ← 推理过程异常
```

### 3.4 服务器内部处理流程

```
┌─────────────────────────────────────────────────────────────┐
│ 收到 JSON ─► 解析 ─► 飞行阶段推断 ─► 构造 Prompt            │
│                                                             │
│ 飞行阶段推断规则（按优先级）:                                 │
│   1. 着陆  : AGL < 50 ft 且 VS < -100 fpm                   │
│   2. 进近  : AGL < 1500 ft 且 VS < -300 fpm                 │
│   3. 滑行  : AGL < 100 ft 且 IAS < 60 kts                  │
│   4. 下降  : VS < -500 fpm                                  │
│   5. 爬升  : VS > 500 fpm 且 ALT < 25000 ft                │
│   6. 巡航  : |VS| < 300 fpm 且 ALT > 25000 ft              │
│   7. 兜底  : 根据 VS 正负判断爬升/下降/巡航                    │
│                                                             │
│ Prompt 构造 (§4.3 模板) ─► Tokenize ─► 逐 token 生成        │
│                                                             │
│ 生成参数:                                                    │
│   max_tokens=64, temperature=0.1, top_p=0.9,                │
│   repetition_penalty=1.1                                    │
│                                                             │
│ 输出: 每个 token ─► {"token": "..."} ─► WebSocket 帧         │
│       推理结束  ─► {"done": true}        ─► WebSocket 帧     │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 请求取消机制

如果服务器正在生成上一帧的响应，而客户端又推送了新一帧数据：

1. 服务器立即设置取消信号
2. 当前生成循环在下一个 token 之前检测到取消信号，停止生成
3. **不会**为旧请求发送 `{"done": true}`
4. 新请求的 prompt 构建和推理由此开始

这确保了：
- 服务器始终处理**最新**的飞行状态
- 客户端不会收到过期建议
- GPU 资源不被浪费在过时的推理上

## 四、开发机端集成指南

### 4.1 连接生命周期

```
启动 ─► TCP 非阻塞连接 ─► WebSocket 握手 ─► 推送首帧 ─► 持续交互 ─► 断开 ─► 重连
```

### 4.2 首帧推送时机

WebSocket 握手成功后，**立即**推送当前完整飞行状态 JSON。然后按阈值触发后续推送。

### 4.3 推送周期 / 轮询

```
                    ┌──── 200ms 最小间隔 ────┐
                    │                        │
  [状态变化>阈值] ──► 等待距上次推送≥200ms ──► 发送 JSON
                    │                        │
                    └── 未到则推迟至满200ms ──┘

  连续变化参数 (alt, ias, vs, heading):
    → 每 200ms 评估一次，超阈值才推送
    → 如果参数稳定（如巡航），长时间不推送

  离散变化参数 (gear, AP, warn/caution):
    → 立即推送（不需要等200ms间隔）
    → 但后续正常推送仍需遵守200ms间隔
```

**Python 伪代码**：

```python
last_push_time = 0
MIN_INTERVAL_MS = 200

def should_push(new_state, old_state):
    """检查是否有参数变化超过阈值"""
    for key, threshold in THRESHOLDS.items():
        if abs(new_state[key] - old_state[key]) > threshold:
            return True
    return False

def push_state(state):
    global last_push_time
    now = time_ms()

    # 离散量立即推送
    if discrete_changed(state):
        send_ws(state)
        last_push_time = now
        return

    # 连续量检查阈值 + 间隔
    if should_push(state, last_state):
        elapsed = now - last_push_time
        if elapsed >= MIN_INTERVAL_MS:
            send_ws(state)
            last_push_time = now
        else:
            # 推迟到满 200ms 再发
            schedule_push_at(last_push_time + MIN_INTERVAL_MS, state)
```

**C 端实现要点**（`src/ai/ai_advisor.c`）：

```c
#define MIN_PUSH_INTERVAL_MS  200   // 最小推送间隔
#define MAX_PUSH_INTERVAL_MS  5000  // 最大静默时间 (5s必须发一次保活)

static uint32_t last_push_tick = 0;

bool ai_advisor_should_push(flight_state_t *new, flight_state_t *old) {
    uint32_t now = get_tick_ms();

    // 离散量变化 → 立即推送
    if (new->gear_down != old->gear_down ||
        new->ap_engaged != old->ap_engaged ||
        new->master_warning != old->master_warning ||
        new->master_caution != old->master_caution) {
        return true;
    }

    // 未到最小间隔 → 不推
    if (now - last_push_tick < MIN_PUSH_INTERVAL_MS) {
        return false;
    }

    // 检查连续量阈值
    if (fabs(new->alt_ft - old->alt_ft) > 50.0f) return true;
    if (fabs(new->ias_kts - old->ias_kts) > 5.0f) return true;
    if (fabs(new->vs_fpm - old->vs_fpm) > 100.0f) return true;
    if (fabs(new->heading - old->heading) > 3.0f) return true;
    // ... (其余参数)

    return false;
}
```

### 4.4 接收与渲染

C 端 WebSocket 回调中：

1. 收到 `{"token": "..."}` → 追加到显示缓冲区
2. 收到 `{"done": true}` → 缓冲区内容提交到 EICAS 显示，清空缓冲
3. 收到 `{"error": "..."}` → 显示 "AI: --"，记录日志
4. 收到 `{"heartbeat": ...}` / Ping → 忽略（自动回复 Pong）

**显示格式**（EICAS 底部）：

```
AI  ADVISORY
  当前巡航状态，一切正常。
```

**超时处理**：如 2 秒内未收到 `{"done": true}`，显示上一次的建议并标记 "(过期)"。

### 4.5 重连策略

```
断开 ─► 等待 2s ─► 尝试重连
              ├── 成功 → 推送当前状态
              └── 失败 → 等待 4s → 8s → 16s → 30s（封顶）

优雅降级:
  服务器不可用时 → cockpit.exe 正常运行
  EICAS 底部不显示 AI 行（或显示 "AI: 离线"）
```
