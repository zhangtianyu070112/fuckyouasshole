## 一、Motivation：为什么需要数据飞轮

### 1.1 当前 AI Copilot 的局限

当前 AI 副驾驶（Qwen2.5-3B Full Fine-tuning）基于**静态训练数据**工作：

```
训练数据 v2（37.7k 条）→ Full FT → 部署模型 → 推理
                          ↑
                    数据冻结，无法适应新场景
```

问题：
- **分布外泛化差**：训练数据覆盖的告警场景有限，实际使用中会遇到未见过的飞行状态组合
- **无法从错误中学习**：模型给出不当建议后，没有机制将反馈回传
- **数据分布漂移**：模拟器版本更新、新机场加入后，原有训练分布不再准确
- **长尾场景缺失**：稀有告警（如 WINDSHEAR、FIRE_CARGO）在 37.7k 条数据中占比极低

### 1.2 数据飞轮的核心思想

```
用户驾驶操作 → 截取关键帧 → 自动/人工标注 → 增量微调 → 模型改进 → 更好的辅助体验
     ↑                                                                              ↓
     └────────────────────────── 正向循环 ───────────────────────────────────────────┘
```

**每次使用都在产生训练数据**。随着使用时间增长，模型持续改进，形成自我增强的正反馈循环。

## 二、三层截取策略（Three-Layer Capture Strategy）

### 2.1 L1 — 告警自动触发（零人工成本）

**原理**：39 类告警系统本身就是高精度标注器。告警触发 = 天然训练标签。

```
告警触发 → 截取前 30s + 后 10s 飞行状态快照 → 自动生成标签
```

| 告警 | 自动标签 |
|---|---|
| `TERRAIN_PULL_UP` | "GPWS触发地形避让警告，需立即执行复飞程序" |
| `WINDSHEAR_WARNING` | "风切变遭遇，飞行员需增加推力保持姿态" |
| `AP_DISCONNECT` | "自动驾驶断开，需手动接管飞机" |
| `STALL` | "失速警告触发，需推杆降低迎角并增加推力" |
| `ENG_OVERHEAT` | "发动机超温，需监控EGT并考虑减推力" |

**特点**：
- 自动标注，零人工介入
- 标签质量高（告警阈值经过航空工程验证）
- 覆盖所有 39 种告警类型
- L1 同类告警 5 秒冷却期，防止重复截取

**代码路径**：`flywheel_alert_trigger()` ← `alert_system.c` 告警触发点

### 2.2 L2 — 模型低置信度截取（半自动）

**原理**：当模型输出 softmax 熵高（top-1 和 top-2 概率接近），说明模型不确定，这正是最有价值的训练数据。

```
模型输出低置信度(entropy > threshold) → 标记为"不确定样本" → 推送人工审核队列
```

**判断条件**：
- Softmax top-1 probability < 0.6，或
- Top-1 与 top-2 概率差 < 0.2

**特点**：
- 直接针对模型弱点采样（hard example mining）
- 需要轻量人工审核（确认/修正 AI 建议）
- 每条 L2 样本的修正都直接改善模型的薄弱区域
- 预期占总数据量的 15-25%

**代码路径**：`flywheel_confidence_trigger()` ← `ai_advisor.c` 推理响应解析点

### 2.3 L3 — 随机分层采样（定期自动）

**原理**：L1/L2 偏向异常场景，L3 通过随机采样确保**正常飞行阶段**的数据覆盖，防止分布漂移。

```
定时器触发（每 120s）→ 截取最近 10s 正常飞行快照 → 阶段自动标注
```

**分层策略**（按飞行阶段）：
| 阶段 | 占比目标 | 原因 |
|---|---|---|
| 巡航 | 40% | 最长飞行阶段，但告警最少 |
| 爬升 | 15% | 推力变化大，发动机参数关键 |
| 下降 | 15% | 能量管理场景丰富 |
| 进近 | 15% | 告警集中区，但正常进近也需要覆盖 |
| 起飞/着陆 | 15% | 最关键的飞行阶段 |

**特点**：
- 完全自动，无需人工
- 防止模型"遗忘"正常飞行模式（catastrophic forgetting 防御）
- 数据量可控（每小时 ~30 条 L3 样本）

**代码路径**：`flywheel_set_random_interval()` + `flywheel_update()` 内部定时器

### 2.4 三层对比总结

| 维度 | L1 告警触发 | L2 低置信度 | L3 随机采样 |
|---|---|---|---|
| **触发条件** | 39 种告警激活 | 模型 softmax 熵高 | 定时器（120s） |
| **截取窗口** | 前 30s + 后 10s | 前 15s + 后 5s | 前 10s |
| **标注方式** | 自动（告警名→标签） | 人工审核 | 自动（阶段名→标签） |
| **人工成本** | 零 | 低（确认/修正） | 零 |
| **数据价值** | 高（异常场景） | 最高（模型弱点） | 中（分布覆盖） |
| **预计占比** | 55-65% | 15-25% | 20-30% |

### 3.2 环形缓冲区（Ring Buffer）设计

```
Ring Buffer: capacity = 1200 frames, stride = 1 (每帧写入)

  ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
  │ 0 │ 1 │ 2 │...│599│600│601│...│1197│1198│1199│
  └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴────┴────┴────┘
                    ▲                   ▲
                    │ trigger_frame_idx │ ring_head (写入位置)
                    │ (告警触发点)        │
                    │                   │
            ┌───────┘                   └──────────┐
            │ pre_window: 30s                       │ post_window: 10s
            │ ← 往回 600 帧                          │ → 往前 200 帧
            ▼                                       ▼
    ┌──────────────┐                       ┌──────────────┐
    │ 截取窗口 800 帧                        │ Capture 最终化 │
    │ (40s 总长)                             │ → JSON 文件   │
    └──────────────┘                       └──────────────┘
```

**关键设计决策**：
- **延迟最终化**：L1/L2 触发时仅记录触发帧索引，继续收集 post-trigger 帧，等 deadline 到达后再从环形缓冲区拷贝帧数据
- **JSON 在最终化时构建**：避免保存期间的重复序列化开销
- **内存预算**：1200 帧 × ~600 字节/帧 ≈ 720 KB（环形缓冲区），每个 Capture JSON 最大 4 MB

## 五、增量微调策略

### 5.1 触发条件

不采用"每条新样本立即微调"的策略，而是累积批量：

| 触发条件 | 阈值 | 说明 |
|---|---|---|
| 样本累积 | 500 条新标注 | L1+L2+L3 合计 |
| 时间累积 | 每 24 小时 | 即使未满 500 条也执行 |
| 质量检查 | L1 自动标签准确率 > 95% | 通过人工抽检 50 条验证 |

### 5.2 微调流程

```
Step 1: flywheel_flush() → data/flywheel/*.json
Step 2: flywheel_to_sharegpt.py → data/new_samples.jsonl
Step 3: 合并 scripts/data/training_data.jsonl + new → merged.jsonl
Step 4: 去重（MD5 hash of flight_state + label）
Step 5: QLoRA 增量微调（r=8, alpha=16, 1 epoch, ~15分钟/500条）
Step 6: LoRA 权重合并到基座模型
Step 7: A/B 测试（旧模型 vs 新模型，各推理 50 条测试用例）
Step 8: 新模型胜出 → 替换部署；否则回滚并标记异常样本
```

### 5.3 安全回滚机制

```
旧模型 (v2.0) ──────────────────────────────▶ 生产环境
                    │
新模型 (v3.0) ──▶ A/B 测试 ──┬── 胜出 ──▶ 替换 v2.0
                              │
                              └── 败退 ──▶ 标记异常样本 → 人工审查
                                           → 回滚到 v2.0
```

## 八、答辩要点（Q&A 预期）

### Q1: 数据飞轮和普通的数据采集有什么区别？

普通数据采集 = 单向流水线（采集→存储→结束）。数据飞轮 = 闭环系统（采集→标注→训练→部署→更多采集）。核心差异在于**自动化闭环**和**模型改进的正反馈**。

### Q2: 自动标签（L1）的质量如何保证？

三重保障：
1. **告警阈值经过航空工程验证**（GPWS 标准如 TSO-C151c），不是拍脑袋定的
2. **定期人工抽检**（每 500 条 L1 样本抽查 50 条，准确率 < 95% 则暂停自动标注）
3. **反向验证**：如果加入某类 L1 样本后模型在该场景下反而变差，说明标签质量有问题

### Q3: 环形缓冲区会不会丢数据？

会 — 这是刻意的设计取舍。环形缓冲区大小 = 60s × 20Hz = 1200 帧。如果 L1 触发时 pre_window=30s，需要 600 帧历史数据。1200 > 600，有足够余量。更早的数据被覆盖是预期行为（不需要 2 分钟前的数据来标注一个刚发生的告警）。

### Q4: 为什么不直接在 Python 层做数据采集？

1. **延迟**：Python 层通过 HTTP 轮询获取飞行数据，采样率远低于 C 层直接访问
2. **可靠性**：UDP 可能丢包，C 层环形缓冲区保证至少保留最新 60s 数据
3. **解耦**：飞轮模块独立于推理服务器，即使推理宕机，数据采集继续运行

### Q5: 存储成本如何控制？

- 环形缓冲区：720 KB（常驻内存，不落盘）
- 每条 JSON 样本：~500 KB（L1 800 帧 × 500B）到 ~50 KB（L3 200 帧 × 250B）
- 按每小时 30 条 L3 + 10 条 L1 + 5 条 L2 ≈ 45 条/小时 ≈ 15 MB/小时
- 一天 8 小时模拟 ≈ 120 MB — 完全可接受
- 增量微调后，原始 JSON 可压缩归档或删除

### Q6: 飞轮数据和现有 37.7k 训练数据的关系？

37.7k 训练数据是**冷启动基座**（覆盖 23 条告警规则的组合场景），飞轮数据是**持续改进增量**（覆盖实际使用中出现的边缘 case）。两者互补：
- 基座数据：广度覆盖（组合 23 条规则 × 6 种表述变体 × 多种飞行阶段）
- 飞轮数据：深度覆盖（真实飞行片段 × 精确时间序列 × 上下文连续性）

## 九、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| L1 自动标签错误 | 低 | 中 | 阈值经过验证 + 定期抽检 + 反向验证 |
| L2 低置信度样本积压 | 中 | 低 | 标注队列最大 200 条，超限自动淘汰最旧样本 |
| 环形缓冲区内存溢出 | 极低 | 高 | 编译期常量检查（`static_assert`）+ 运行时边界检查 |
| 增量微调导致模型退化 | 中 | 高 | A/B 测试门控 + 自动回滚机制 |
| 磁盘写满 | 低 | 中 | 监控磁盘使用率 + 自动清理 30 天前的原始 JSON |
