# 航空驾驶舱模拟控制系统 — 项目需求报告

> 开题阶段文档 · 2026 年 6 月下旬
> 技术细节已在开发过程中逐步明确并验证

---

## 一、项目背景

飞行模拟器这东西，全动模拟机一台几千万，维护成本也高，基本上只有航空公司和专业训练机构才用得起。航校的学生、航空爱好者想摸一摸驾驶舱，机会很少。但另一方面，现在一台普通 PC 的性能已经完全够跑一个桌面级的仪表仿真了——CPU 渲染 60 帧没问题，网络也能做实时数据对接。

所以这个项目的想法很简单：**用纯 C + SDL2 在 Windows 上搭一套桌面级飞行驾驶舱模拟系统。** 不用依赖任何昂贵的硬件，不需要专业模拟器软件，一个可执行文件拉起来就能看到 PFD、ND、EICAS 这些仪表在跑，FMC 能编辑航路，数据能从 X-Plane 12 实时喂进来。

项目最终交付的是一个**双座驾驶舱布局**——机长侧和副驾驶侧各一套 PFD + ND，中间共享 EICAS 和 FMC，一共 6 块仪表同时渲染。分辨率适配 2560×1600 全屏桌面。

## 二、项目目标

### 2.1 知识目标

说白了就是在实际项目里把 C 语言的高级特性用起来。不是写那种几百行的课后作业，是真要面对一个 ~19,500 行的系统：

- **指针和内存管理**：动态分配、手动释放、不能有泄漏。每个 malloc/calloc 有对应的 free，所有结构体用 memset 初始化避免未定义行为。
- **结构体设计**：FlightData（~60 个飞行参数字段）、FMCState（航路计划+CDU 状态）、RouteGraph（图节点+边数组+A* 工作状态）——字段名要对得上实际含义，嵌套层次清晰。
- **文件 I/O**：INI 配置文件解析、OpenFlights airports.dat（7698 机场→筛选 179 个美国+中国活跃机场）、earth_fix.dat（~198k 航路点）+ earth_nav.dat（~37k 导航台）加载。
- **SDL2 图形渲染**：窗口管理、渲染管线、B612+B612Mono TTF 字体纹理缓存（1024 条上限）、阿里普惠体中文渲染。SDL2_gfx 画抗锯齿圆和线条。8K 地球纹理贴图替代瓦片方案。
- **仪表显示逻辑**：PFD 的姿态指示器+空速带+高度带+航向带+垂直速度指示器+N1 推力指示器+FMA 4 列布局；ND 的四种模式（VOR/MAP/APP/PLN）+ 空间哈希实时导航叠加；EICAS 的双发 180° 半圆弧表盘+燃油三油箱显示+CAS 告警灯面板。
- **FMC 飞行管理计算机**：10 页 CDU 界面（IDENT/RTE/LEGS/DEP ARR/CLB/CRZ/DES/PROG/NAV RAD/PERF），完整键盘布局（3×5 功能键+3×4 数字+4×6 字母+编辑键），LSK 行选键实体按钮（CDU 两侧 36px 按钮列带 L1-L6/R1-R6 标注），A* 自动寻路+手动航点编辑。
- **数据结构从零实现**：6 种——哈希表（字符串键，50 万桶，~0.47 负载因子）、AVL 树（字母序/航路序排序切换）、双向链表（FMC 消息历史+EICAS 告警日志）、空间哈希网格（1°纬度带×动态经度扇区，2003 桶，64-bit 打包键）、链式前向星图（3 并行数组，O(1)边遍历）、二叉最小堆（A* 开放集）。
- **UDP 网络编程**：五层协议栈——RPOS（位置请求）+ DSEL（27 个数据组订阅）+ DATA（N×36B 多组打包解析）+ DREF（509B 固定包写入）+ RREF（25 条告警 DREF 订阅，413B 固定请求包）。支持 CMND 命令发送（FMC 键盘镜像到 XP12 CDU）。
- **多线程**：渲染主线程 + UDP 接收线程 + 音频回调线程 + 客舱 HTTP 服务线程，四线程并发。mutex 保护共享数据，每个子系统独立锁粒度。

### 2.2 能力目标

- 独立走完**需求分析 → 模块设计 → 编码 → 调试 → 集成测试**的全流程
- 面对"~235,000 条导航数据里怎么在 16ms 内找到飞机周围几百海里的点"这种实际问题，能选对数据结构（空间哈希网格，~100×加速比，~0.5ms 完成单次查询）并实现
- 能把 XP11/XP12 的 UDP 二进制协议从抓包逆向到稳定对接——包括 XP12 变更点（DATA* 0x2A 包标识、DREF 509B 固定包、160+ 数据组索引重排、多组合并解析）
- 60 FPS 稳定渲染，不掉帧不卡顿
- 写清楚文档，代码有注释，模块有边界，编译 `-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Wstrict-prototypes` 零警告

### 2.3 素质目标

- 理解飞 simulation 在航空安全里的实际意义——GPWS 告警不是闹着玩的，是真能救命的东西
- 培养把"业务需求"翻译成"数据结构+算法"的能力
- 对实时系统、图形渲染、航空电子保持好奇和探索欲
- **安全关键系统的分层设计思维**——确定性系统（C 层硬告警）的优先级永远高于概率性系统（LLM 建议）

## 三、功能需求

### 3.1 仪表显示系统

#### 3.1.1 PFD — 主飞行显示器

驾驶舱里最核心的一块仪表，放在机长侧正前方。实现内容：

- **姿态指示器**：天地背景（蓝上棕下），基于地平线两端点的**逐行交叉积**判定算法填充——每行测试左右两端点是否同侧，同侧整行单色，异侧从交点分割。保证天地填充边界和地平线数学上完全重合，没有黑缝。
- **俯仰刻度**：±90° 范围，每 10° 刻度线+标注，2.5° 小刻度。地平线随俯仰角连续滚动。
- **滚转刻度**：顶部弧形刻度带，±60° 范围，10°/20°/30°/45°/60° 刻度标记。
- **空速带**：左侧垂直刻度带，`px_per_kt = 5.0`，当前空速在中间。超速红黑区 + 失速警告区。刻度用 `(tick_value - current_float) * px_per_kt` 逐像素浮动滚动——不是 int 截断。
- **高度带**：右侧垂直刻度带，`px_per_ft = 0.6`。目标高度 bug（洋红色菱形）。
- **航向带**：底部水平弧带，`px_per_deg = 5.0`。N/E/S/W 用 TTF 大字加方向角小字。5° 小刻度+10° 大刻度+30° 标注。
- **VSI 垂直速度指示器**：7% 宽度完整垂直刻度条，±6000 fpm 量程，每 1000 fpm 刻度线加偶数标注。青色三角指针加绿色数字读数，加 smooth_vs 平滑变量防跳变。
- **N1 推力指示器**：FMA 下方，双发水平条形图（灰底+绿色/琥珀填充+红色 100% 参考线+数字%），加 smooth_n1[2] 平滑。
- **FMA（Flight Mode Annunciator）**：4 列布局——A/THR|ROLL|PITCH|AP。激活时绿/品红文字加暗色背景框，未激活灰色。数据驱动动态模式切换（HDG|ALT / HDG|V/S / MANUAL）。
- **马赫数**：SPD 标签下方，M > 0.4 时显示。
- **无线电高度**：ALT 标签下方，AGL < 2500 ft 时绿色显示。
- **AP 航向/高度/速度 bug**：洋红色三角/菱形标记。
- **主警告/注意灯**：FMA 右侧红闪/琥珀闪覆盖层，600 ms 周期。

#### 3.1.2 ND — 导航显示器

放在 PFD 旁边，四种显示模式（按 m 键循环切换）：

**MAP 模式（默认）**：
- 罗盘玫瑰 + 范围环，磁航向基准
- 空间哈希网格实时查询飞机周围的导航台和航路点——1° 纬度带分区，`1.0/cos(lat)` 动态经度扇区宽度，5×5 邻域搜索按 cosine 航向相似度排序（**航向优先策略**，前方网格先搜）
- 航路点青色菱形、机场品红方块、航线绿色连线、活跃航段品红高亮
- 量程 7 档：5/10/20/40/80/160/320 NM（按 +/- 键切换）
- 顶部信息栏（38px 半透明暗底+分隔线）：TRK|MAG（GPS 位置差实时计算地面航迹角，GS<30 节回退磁航向），目标航路信息（下一航点标识品红+距离 NM 白+ETA HH:MM 绿），风速箭头+数值标签

**VOR 模式**：
- 全罗盘 + VOR 航道指针 + CDI 偏航指示 + DME 距离
- VOR 径向线几何计算（飞机位置→台站方位角）

**APP 模式**：
- ILS 仪表进近：LOC 航向道偏差（水平）+ GS 下滑道偏差（垂直）+ 航向刻度带
- 两种偏差量同时显示

**PLN 模式**：
- 正北朝上静态计划视图，显示完整飞行计划航路
- 坐标系变换（磁北旋转→正北朝上）

ND 快捷键：w（航路点显隐）、a（机场显隐）、r（航线显隐）、v（VOR-NDB 显隐）、c（航道显隐）。

**导航数据库关键数据**：加载 earth_fix.dat（~198k 航路点）+ earth_nav.dat（~37k VOR/NDB/机场），空间哈希网格总计 ~235k 条目。单次邻近查询 ~0.5 ms（vs. 线性扫描 ~50 ms），加速比 **~100×**。

#### 3.1.3 EICAS — 发动机指示和机组告警系统

**发动机显示**：
- **N1 双发半圆弧表盘**：180° 圆弧（9:00→3:00），灰色轨道+白色刻度/指针，超标变红
- **EGT 双发半圆弧表盘**：同上布局，红线 950°C
- **FF 燃油流量**：圆弧下方数字显示+PPH 单位
- **TAT 总温**：标题栏右侧，含 ram rise 修正 `(TAS/100)² × 0.55`
- **ENG1/ENG2 参数表**：N1%/EGT°/N2% 三行两列并排
- **燃油系统**：左翼/中央/右翼三油箱独立显示（46%/8%/46% 分配），L/CTR/R 绿色标签+青色标签，TOTAL 汇总行

**CAS 告警灯面板**：
- 50×28 px 方形灯，4 列紧凑布局
- 12 个系统告警灯数据驱动，实时响应

**音频告警系统**：
- **39 种告警类型**，按数据源分三类：UDP DATA Group（飞行参数阈值检测）、DREF RREF（25 条告警状态硬信号订阅）、Derived（从已有参数计算）
- GPWS 13 种：PULL UP / WINDSHEAR / TERRAIN / SINK RATE / TOO LOW GEAR / TOO LOW FLAPS / GLIDESLOPE / BANK ANGLE / OVERSPEED / MINIMUMS + 高度自动喊话 10 档（500/400/300/200/100/50/40/30/20/10 ft）
- 系统告警 10 种 + RREF 硬信号 25 种
- SDL2 音频回调实时合成波形（方波/正弦波/锯齿波），零外部音频文件依赖
- 优先级抢占 + 冷却计时器防重复
- 无音频设备时优雅降级
- 架构预留 WAV PCM 真人语音替换接口

**告警安全架构（Route A）**：
```
Layer 1: C 层硬告警（Hard Alerts）
  ├─ 飞行参数阈值检测（alert_system.c）
  ├─ XP12 RREF 标志位（25 条 DREF 订阅）
  └─ 优先级抢占 + 冷却计时
  → 输出: 音频播报 + EICAS CAS 告警灯

Layer 2: AI 软建议（Soft Advisory）
  └─ Qwen2.5-3B 态势感知
  → 输出: EICAS 文本建议行
  → 不可绕过 Layer 1 的硬告警
```

C 层硬告警是不可绕过的最后防线，AI 仅为补充建议。这是安全关键系统的核心设计原则。

### 3.2 FMC — 飞行管理计算机

FMC 是整个模拟系统里逻辑最复杂的模块。做成一个完整的 CDU 交互界面：

**10 页 CDU 界面**：

| 页面 | 内容 |
|------|------|
| IDENT | 飞机型号、发动机型号、导航数据库有效期 |
| RTE | 输入 ORIGIN/DEST ICAO → ACTIVATE 触发 A\* 自动寻路 → 填充 FlightPlan |
| LEGS | 航路点列表+累计距离，支持 LSK 手动插入/删除航点，PGUP/PGDN 翻页+5 step |
| DEP ARR | 三级状态机选 SID/STAR+跑道（数据库驱动，256 机场×24 跑道×16 SID） |
| CLB | 巡航高度 FL350、目标速度 M.78/280 KTS、10000 ft 以下 250 KTS 速度限制 |
| CRZ | 巡航高度、目标马赫数、最优/最大高度 |
| DES | 终点高度、目标速度、3.0°下降角、过渡高度 18000 ft |
| PROG | 已飞距离、剩余距离、预计到达时间、燃油估算 |
| NAV RAD | COM1/COM2/NAV1/NAV2/XPDR/DME 频率管理 |
| PERF | 性能参数（成本指数、备降机场等） |

**CDU 按键布局**：
- 功能键 3×5（INIT REF/RTE/CLB/CRZ/DES、DIR INTC/LEGS/DEP ARR/HOLD/PROG、FIX/NAV RAD/PREV PAGE/NEXT PAGE/PERF）
- EXEC 独立键（CDU 屏下方右对齐，宽度 30%，绿色内框加亮底色）
- 字母 4×6（A-E/F-J/K-O/P-T/U-Y，Z 单独一行）
- 数字 3×4（浅色底区分）
- 编辑键（CLR/DEL/SP，右侧单列琥珀暖底加琥珀边框）
- 便签本（Scratchpad）底部输入行，左对齐加闪烁光标

**LSK 行选键**：CDU 屏幕两侧各 36 px 按钮列，L1-L6/R1-R6 实体可见按钮 + 琥珀色高亮边框。Hover 时白色高亮，Click 时灰色闪烁 200 ms。FMC 字母数字键输入被禁掉（仅保留功能键+鼠标点击 CDU 面板），字母键全部归 ND 快捷键使用。

**航路规划 — A* 寻路引擎**：
- **图结构**：链式前向星（3 并行数组 `head[]/to[]/next[]` + 权重数组 `dist_nm[]`），O(1) 边遍历，比链表邻接表 cache 友好，比邻接矩阵节省 99.9% 空间
- **启发函数**：Haversine 大圆距离（球面余弦定律+Haversine 稳定形式），满足可纳性，保证全局最优解
- **方向惩罚项**：每度偏差 +0.5 NM 惩罚，优先展开朝向终点的边
- **开放集**：二叉最小堆（`heap[]` + `sift_up/sift_down` + `heap_pos[]` 反向索引），支持 O(log N) push/pop + O(1) 成员判断 + O(log N) decrease-key
- **闭合集**：世代标记 `visit_token`——每次搜索前递增一个整数，O(1) 重置，避免 235k 节点的 memset
- **图构建**：空间哈希邻近查询（每个节点 4 方向各查 32 条邻近结果，取前 20 条）+ 机场↔航路点直接连接（1.5° 边界框过滤，300 NM 范围内，每机场最多 8 条）
- **图规模**：~235k 节点 + 数十万条有向边，179 机场覆盖美国本土 48 州全部主要枢纽 + 中国 10 个主要机场
- **寻路性能**：1000 节点图北京→上海约 **7.5× 加速**（vs. Dijkstra），大部分美国本土航线 50 ms 内出结果

**FlightPlan 完整参数**：
- 爬升：目标速度 M.78/280 KTS，10000 ft 以下 250 KTS 限制
- 巡航：目标马赫数 M.78，最优/最大高度
- 下降：目标速度 M.78/280 KTS，3.0° 下降角，过渡高度 18000 ft，单发加速高度

**O(1) 航点查找**：FMC 初始化时构建 50 万桶哈希表（~0.47 负载因子），`ht_get()` 在 235k 条目中 O(1) 定位航点——用户在便签本输入航点标识符后按 LSK 瞬间响应。

### 3.3 飞行数据接入

#### 3.3.1 X-Plane 12 UDP 协议栈

完整实现了五层原生 UDP 协议，不用任何第三方插件：

| 方向 | 协议 | 端口 | 格式 | 说明 |
|------|------|------|------|------|
| XP12→Cockpit | **RPOS** | 49005 | 5B 固定 | 请求位置数据流 |
| XP12→Cockpit | **DSEL** | 49005 | 6B 固定 | 订阅 27 个数据组 |
| XP12→Cockpit | **DATA** | 49005 | N×36B 多组打包 | 实际飞行数据（6233 字节/包） |
| Cockpit→XP12 | **DREF** | 49000 | **509B 固定** | DataRef 写入（float+path 400B null-padded+padding 100B） |
| Cockpit→XP12 | **CMND** | 49000 | ~21B | FMC 键盘镜像到 XP12 CDU |
| Cockpit→XP12 | **RREF** | 49000 | 413B 固定 | 25 条告警 DREF 订阅（freq+request_id+path 400B） |

**XP11→XP12 协议迁移中解决的 7 个问题**：

1. **DATA 包第 5 字节变化**：XP11 用 `0x00`（`DATA\0`），XP12 改为 `0x2A`（`DATA*`）。不识别此差异会导致所有 DATA 包被丢弃。
2. **509 字节固定 DREF 包**：XP12 要求统一 509B（vs. XP11 可变长 42-44B），这是为防止缓冲区溢出新增的协议约束。
3. **多组合并解析**：XP12 将所有数据组打包到一个 UDP 包，旧代码只解析第一组→改为循环解析。
4. **数据组索引大重排**：160+ 数据组全部新编号（N1: 26→41, EGT: 34→47, FF: 37→45, AP: 63→108+118），手动逐组测绘，写入 `xplane12_data_groups.json` 映射表。
5. **DREF 粘包**：3 条 DREF 零间隔连续发送导致 XP12 错误切分包边界，每两条间加 `SDL_Delay(15)` 消除。
6. **RREF 订阅系统**：25 条告警状态 DREF 批量订阅，XP12 以 1-4 Hz 主动推送 float。将告警数据采集带宽从轮询 ~24 KB/s 降至推送 ~300 B/s。
7. **单位制统一**：XP12 全部英制（ft, ft/min, lbs/hr），移除所有公制转换系数。N1/N2 发送 0.0-1.0 比率，自动检测 ≤2.0 的值 ×100 转百分比。

**FlightData 容器**：~60 个字段——姿态、空速、马赫数、气压/无线电高度、垂直速度、航向、N1/N2/EGT/FF、滑油、液压、电气、防冰、增压、无线电频率、应答机、告警灯状态等。

#### 3.3.2 模拟数据发生器

## 五、技术架构

### 5.1 核心数据结构总览

| 数据结构 | 实现文件 | 用途 | 关键参数 |
|----------|----------|------|----------|
| 哈希表 | `ds/hash_table.c` | FMC 航点 O(1) 查找、RouteGraph 节点索引 | 50 万桶，DJB2 哈希，~0.47 负载因子 |
| AVL 树 | `ds/avl_tree.c` | FMC LEGS 页排序（字母序/航路序切换） | 平衡因子旋转 |
| 双向链表 | `ds/linked_list.c` | FMC 消息历史（50 条）+ EICAS 告警日志 | 带时间戳 |
| 空间哈希网格 | `ds/spatial_hash.c` | ND 实时导航叠加、RouteGraph 建边 | 1°纬度带×动态经度扇区，2003 桶，64-bit 打包键，航向优先邻域搜索 |
| 链式前向星图 | `data/route_graph.c` | A* 航路寻路 | 3 并行数组（head/to/next）+ 权重数组，O(1) 边遍历 |
| 二叉最小堆 | （route_graph.c 内） | A* 开放集 | heap_pos[] 反向索引支持 decrease-key |

### 5.2 线程模型

```
Main Thread                  UDP Thread               Audio Thread        HTTP Thread
────────────                 ──────────               ────────────        ───────────
SDL Event Loop               recvfrom() loop          SDL2 callback       accept() loop
  → Pump events                → Parse DATA/RREF        → Generate tone     → Serve JSON
  → Snapshot FlightData        → Lock mutex             → Write to buffer   → Lock position
  → Update instruments         → Write FlightData
  → Render instruments         → Unlock mutex
  → Frame cap
```

- 飞行数据通过 mutex 保护的单一流向实现线程安全
- UDP 线程不参与渲染，仅负责数据写入
- 主线程仅短暂持锁做 memcpy 快照（~2 KB，< 1 μs）
- 每个子系统独立 mutex，不存在全局大锁
- 音频线程用原子操作写入 tone 状态，与主线程零冲突
- 死锁预防：遵循单向依赖原则——UDP 线程只写，渲染线程只读

## 六、争议技术决策及实际选择

以下是在开发过程中遇到的关键技术分歧点以及最终的选择和理由：

### 6.1 链式前向星 vs. 链表邻接表 vs. 邻接矩阵

**最终选择**：链式前向星（3 并行数组 `head[]/to[]/next[]` + `dist_nm[]`）。

**理由**：
- 邻接矩阵对 ~235k 节点完全不现实（需要 ~55B 个元素）
- 链表邻接表每条边独立 malloc——50 万条边 = 50 万次 malloc + 50 万次 free + 指针追逐 cache miss
- 链式前向星仅 3 次 realloc 管理全部边，遍历时数组下标连续访问 CPU prefetch 友好，每边仅 12 字节（3 int）+ 4 字节 float = 16 字节 vs. 链表 ~32 字节

### 6.2 空间哈希网格 vs. KD-Tree vs. R-Tree

**最终选择**：空间哈希网格。

**理由**：
- C 语言实现 R-Tree 的节点分裂/MBR 更新复杂且易出错
- 空间哈希 = 哈希表+链表，纯 C 用 `calloc` + `next` 指针即可
- 查询模式是移动窗口（每次查询位置接近上次），空间哈希的固定网格边界天然利用空间局部性
- **领域优化**：航向优先邻域搜索排序——按 cosine 相似度排邻居网格，航线前方的点最先返回。R-Tree/KD-Tree 做不了这种语义排序

### 6.3 Haversine 启发 vs. 欧几里得距离 vs. 曼哈顿距离

**最终选择**：Haversine 大圆距离。

**理由**：
- 欧几里得距离（穿透地球的直线）比真实最短路径更短→仍然可纳但太松→展开过多节点→退化成 Dijkstra
- 曼哈顿距离在球面上不可纳→可能错过最优解
- Haversine 是地球表面两点间最短弧长→最紧的可纳下界→A* 展开节点最少

### 6.4 visit_token 世代标记 vs. memset 清零

**最终选择**：世代标记。

**理由**：
- 235k 个 `int` 的 `memset` = ~1 MB 写入，每次 A* 搜索前都要来一遍
- 世代标记只需递增一个整数+每次写入一个整数，O(1) 重置
- 在频繁路径重算场景（用户改航路点）差距显著

### 6.5 SDL2 实时波形合成 vs. 预录制 WAV 播放

**最终选择**：SDL2 音频回调实时合成 + 预留 WAV 接口。

**理由**：
- 实时合成零外部文件依赖，单可执行文件即可运行
- 39 种告警各有独特波形/频率/节奏设计（6 大类：火灾 5 种 850-1200 Hz 快速脉冲、GPWS 危险 4 种原版方波/滑音、GPWS 警告 5 种 300-1000 Hz 中频脉冲、发动机/系统 6 种、电气/液压 3 种、辅助 6 种）
- 架构预留 `wave_type = WAVE_PCM` 替换接口
