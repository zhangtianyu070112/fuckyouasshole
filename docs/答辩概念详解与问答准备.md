### 🏆 创新点 1：空间哈希网格 — 23.5 万航路点实时查询

#### 概念解释

**空间哈希（Spatial Hash）** 是一种将连续的地理空间离散化为有限网格单元的索引结构。与传统的哈希表不同，它不按 key 的 hash 值分桶，而是按**空间位置**分桶。

#### 我们的设计思维

**问题驱动**：ND 导航显示器需要在每帧（60 FPS）从 ~235,000 个航路点中找出"飞机周围几十海里内"的导航条目。线性扫描 235k 条目每帧不可行（16ms 预算内无法完成），R-Tree/Quadtree 在 C 语言中实现复杂度高。

**三步推理**：
1. **网格划分** → 1° 纬度带（地球 -90°→+90° 分成 180 带），每带内按 `1.0/cos(lat)` 度动态划分经度扇区。这样每个网格约 60×60 NM，保证任何位置只需检查周围 3×3=9 个网格。
2. **航向优先排序** → 不是随意搜索 9 个邻居，而是按 cosine 相似度（飞机航向 vs. 邻居网格中心方向）排序，优先搜索**前方**网格。这意味着航线前方的航路点最先被找到。
3. **64-bit 打包键** → `(lat_band << 32) | lon_sector`，一个 `uint64_t` 即可唯一标识网格，插入和查询都是 O(1) 哈希。

#### 为什么比线性扫描好

| | 线性扫描 | 空间哈希 |
|---|---|---|
| 每次查询 | O(235,000) | O(9 网格 × 平均每格条目数) ≈ O(200-500) |
| 实际耗时 | ~50ms（掉帧） | ~0.5ms（60 FPS 无压力） |
| 加速比 | — | **~100×** |

#### 答辩关键词
> "我们不是简单使用现成的空间索引库，而是针对航空导航场景——飞机总是向前飞——设计了**航向优先的网格搜索策略**，这是通用 GIS 库做不到的领域优化。"

### 🏆 创新点 2：A* 航路寻路 + 链式前向星图

#### 概念解释

**A\* 算法** = Dijkstra + 启发式函数。核心公式：`f(n) = g(n) + h(n)`，其中：
- `g(n)` = 从起点到节点 n 的实际代价
- `h(n)` = 从节点 n 到终点的**估计**代价（启发函数）

**可纳性（Admissibility）**：当 `h(n) ≤ 真实最短距离` 时，A\* 保证找到最优解。我们使用 Haversine 大圆距离作为启发，它是地球表面两点间的最短路径，天然满足可纳性。

**链式前向星（Chain Forward-Star）** = 用三个并行数组（`head[]`, `to[]`, `next[]`）表示图的邻接表，比链表实现更 cache-friendly，内存连续访问。

#### 我们的设计思维

**为什么要自己写 A\* 而不是调用库？**
1. 航路图是**稀疏图**（每个节点平均 3-5 条边），不是网格图
2. 需要**高度层约束过滤**（某些航路仅 FL200+ 可用）
3. 需要输出 `RoutePath` 直接兼容 `FlightPlan` 结构
4. 航空领域需要的距离单位是**海里**，启发函数必须用 Haversine

**三个关键优化**：
| 优化 | 技术 | 效果 |
|---|---|---|
| 二叉最小堆 | `heap[]` + `sift_up/sift_down` | 开放集操作 O(log n) |
| 世代标记闭合集 | `visit_token` 而非 `memset` | O(1) 重置而非 O(n) |
| Haversine 启发 | 大圆距离（球面余弦定律 + Haversine 稳定形式） | 紧下界，保证最优 |

#### 答辩关键词
> "A\* 的核心优势在于**可纳启发函数**——我们使用大圆距离作为下界估计，数学上保证了在任何航路图中都能找到**全局最短路径**。同时链式前向星的内存布局让边遍历对 CPU 缓存友好。"

### 🏆 创新点 3：X-Plane 12 双向原生 UDP 通信

#### 概念解释

X-Plane 的 UDP 协议栈有五层：

| 协议 | 方向 | 用途 |
|---|---|---|
| **RPOS** | XP → 客户端 | 请求位置数据流 |
| **DSEL** | 客户端 → XP | 选择要订阅的数据组 |
| **DATA** | XP → 客户端 | 实际飞行数据（N×36 字节） |
| **DREF** | 客户端 → XP | 写入 DataRef 变量 |
| **CMND** | 客户端 → XP | 发送命令（如 `sim/FMS/key_A`） |
| **RREF** | 双向 | 请求 + 接收 DataRef 值（类似 DREF 的订阅版） |

#### 我们的设计思维

**为什么不直接用 NASA X-Plane Connect 库？**
- XPC 是第三方插件，需要额外安装
- XPC 不支持 XP12 的新数据组编号
- 我们的目标是**零依赖独立运行**

**XP11 → XP12 迁移中的关键发现**（这是答辩中展示工程能力的好素材）：

1. **"DATA\0" → "DATA\*"**：XP12 在第 5 字节用 `0x2A`（`*`）而非 `0x00`（`\0`）作为包分隔符
2. **509 字节固定 DREF 包**：XP11 的 DREF 是可变长（42-44B），XP12 改为 509B 固定长度（float 4B + path 400B null-padded + padding 100B）
3. **多组合并**：XP12 将所有数据组合并到一个 UDP 包发送，旧代码只解析第一组
4. **数据组索引大重排**：XP12 的 160+ 数据组使用全新编号，N1 从 group 26→41，EGT 从 34→47

#### 双向通信的工程价值

大多数实训项目只做**数据接收**（XP→模拟器），我们实现了**反向控制**：
- DREF 写入：FMC 激活航路后，自动设置 XP12 自动驾驶的巡航高度/速度/航向
- CMND 命令：CDU 上每一次按键都镜像到 XP12 的 CDU，实现真正的双向操作

#### 答辩关键词
> "我们从**协议逆向工程**入手，通过抓包对比 XP11 和 XP12 的差异，解决了 7 个阶段的技术问题——这体现了我们在**底层网络协议层面的工程能力**，不是简单地调用现成 SDK。"

### 🏆 创新点 4：39 种 GPWS + 系统告警 + 实时音频合成

#### 概念解释

**GPWS（Ground Proximity Warning System，近地告警系统）** 是民航安全的关键系统。真实 B737 的 GPWS 包含多种模式：

| 模式 | 告警 | 触发条件 |
|---|---|---|
| Mode 1 | SINK RATE | 下降率过大 |
| Mode 2 | TERRAIN | 地形接近率过高 |
| Mode 3 | DON'T SINK | 起飞后高度损失 |
| Mode 4 | TOO LOW GEAR/FLAPS | 起落架/襟翼未放下 |
| Mode 5 | GLIDESLOPE | ILS 下滑道偏离 |
| Mode 6 | BANK ANGLE | 坡度角过大 |
| Mode 7 | WINDSHEAR | 风切变 |

**RREF（Request DataRef）** = X-Plane 的 DataRef 订阅机制：客户端发送 `RREF\0` 包订阅指定 DataRef 路径，XP 以配置的频率回推该值。

#### 我们的设计思维

**Route A 架构决策**（这是答辩中最能展示系统思维的部分）：

```
┌──────────────────────────────────────────────────┐
│              告警检测双层架构                       │
├──────────────────────────────────────────────────┤
│  Layer 1: C 层硬告警 (Hard Alerts)                │
│  ├─ 飞行参数阈值检测 (alert_system.c)              │
│  │  AGL < 100ft + VS < -2000fpm → PULL UP        │
│  ├─ XP12 RREF 标志位 (25 条 DREF 订阅)            │
│  │  sim/cockpit/warnings/... → 直接读取告警灯      │
│  └─ 优先级抢占 + 冷却计时                         │
│  → 输出: 音频播报 + EICAS CAS 告警灯              │
├──────────────────────────────────────────────────┤
│  Layer 2: AI 软建议 (Soft Advisory)               │
│  └─ ChatGLM3-6B 态势感知                           │
│  → 输出: EICAS 文本建议行                          │
│  → 不可绕过 Layer 1 的硬告警                       │
└──────────────────────────────────────────────────┘
```

**安全设计的核心逻辑**：GPWS 是**生命安全关键系统**，不能依赖 AI 模型。LLM 可能产生幻觉（hallucination），所以 C 层硬告警是**不可绕过的最后防线**，LLM 仅为**补充建议**。

**音频合成的工程选择**：
- 使用 SDL2 音频回调 + 实时波形合成（方波/正弦波/锯齿波），而非播放 WAV 文件
- 优势：零外部音频文件依赖，单个可执行文件即可运行
- 架构预留了 WAV 真人语音替换接口（`tone_generator` 结构体中的 `wave_type` 可扩展为 `WAVE_PCM`）

#### 答辩关键词
> "我们采用了**安全关键系统的分层架构**——C 层硬告警是最后防线，AI 建议是辅助增强。这种架构体现了航空电子系统设计的核心原则：**确定性系统（C）的优先级永远高于概率性系统（LLM）**。"

### 🏆 创新点 5：AI 副驾驶系统

#### 概念解释

**QLoRA（Quantized Low-Rank Adaptation）** = 在 4-bit 量化的基座模型上训练低秩适配矩阵。
- **量化**：模型权重从 FP16（2 字节/参数）压缩到 4-bit（0.5 字节/参数），6B 模型从 12GB → ~3.5GB
- **LoRA**：不修改原始权重，只训练两个小矩阵 A 和 B（r=16, α=32），可训练参数仅 ~0.1% 的原始模型
- **效果**：在 2×RTX 3090 (24GB×2) 上即可微调 6B 模型，无需 A100

**GGUF** = llama.cpp 的模型格式，支持 CPU+GPU 混合推理，Q4_K_M 量化后 ~4GB，可在单卡上全 GPU offload。

**WebSocket（RFC 6455）** = 全双工持久连接协议。相比 HTTP 请求-响应，WebSocket 的优势在于：
- 握手后保持 TCP 连接，无需每次重建
- 服务端可主动推送（streaming token）
- 头部开销极小（2-14 字节 vs HTTP 的数百字节）

#### 我们的设计思维

**为什么是 WebSocket 而不是 HTTP？**
1. LLM 推理是**流式生成**（token-by-token），HTTP 长轮询效率低
2. 飞行状态推送需要**事件驱动**（参数变化才发送），WebSocket 天然支持
3. 首 token 延迟目标 <60ms，HTTP 的请求-响应往返开销不可接受

**为什么是 C 语言写 WebSocket 客户端？**
- 项目中已有 Winsock2 依赖（UDP 通信）
- 不需要引入 libwebsockets 等第三方库
- RFC 6455 协议本身足够简洁：握手 = HTTP Upgrade + Base64 SHA1 密钥；数据帧 = FIN+opcode+MASK+payload_len+masking_key+payload

**数据生成的工程考量**：
- 不是手工标注数据，而是用 **Python 飞行剖面模拟器 + 规则模板** 自动生成 37,700 条训练样本
- C 层的 `mock_data.c` 因垂直速度严重失真（15,800 fpm vs 真实 2,500 fpm）被 Python 重写替代
- 23 条规则模板从 `alert_system.c` 精确提取阈值，保证训练数据与 C 层逻辑一致

#### 答辩关键词
> "AI 副驾驶的创新点不在模型本身，而在于**系统工程集成**——我们实现了从 C 语言 cockpit → WebSocket → llama.cpp → GGUF 模型 → 流式推理 → C 语言显示的**完整端到端管线**，包括从零手写 RFC 6455 客户端。"

### 🏆 创新点 6：ND 四模式导航显示器

#### 概念解释

| 模式 | 航空术语 | 显示内容 | 使用场景 |
|---|---|---|---|
| **MAP** | 地图模式 | 罗盘玫瑰 + 航路 + 导航台 + 气象 | 航路飞行（默认） |
| **VOR** | 全向信标模式 | 全罗盘 + VOR 航道指针 + CDI 偏航指示 + DME 距离 | VOR 导航/进近 |
| **APP** | 进近模式 | LOC 航向道偏差 + GS 下滑道偏差 + 航向刻度带 | ILS 仪表进近 |
| **PLN** | 计划模式 | 正北朝上 + 完整航路静态视图 | 地面航前准备 |

**CDI（Course Deviation Indicator）** = 偏航指示器，显示飞机偏离预定航道的角度/距离
**ILS（Instrument Landing System）** = 仪表着陆系统，由 LOC（水平引导）和 GS（垂直引导）组成

#### 我们的设计思维

ND 的核心技术挑战是 **Mode-Specific Rendering**：
1. MAP 模式需要**空间哈希实时查询**（23.5 万条目的邻近搜索）
2. VOR 模式需要**VOR 径向线计算**（飞机位置 → VOR 台站的方位角）
3. APP 模式需要**两种偏差量同时显示**（LOC 水平 + GS 垂直）
4. PLN 模式需要**坐标系变换**（磁北旋转 → 正北朝上）

每个模式的渲染逻辑完全不同，使用 `nd_mode` 枚举 + `switch` 分支实现。

#### 答辩关键词
> "ND 四种模式不是简单的显隐切换，每种模式有**完全独立的渲染逻辑和数据源**——MAP 用空间哈希、VOR 用无线电信标几何、APP 用 ILS 偏差——这需要深入的航空导航知识支撑。"

### 🏆 创新点 7：PFD 全浮动连续滚动刻度

#### 概念解释

**驾驶舱仪表的核心设计原则**：信息必须**无跳变、连续平滑**呈现。真实机械陀螺仪/空速表的指针是连续运动的。

#### 我们的设计思维

**Bug 根因**：旧代码用 `int ias = (int)fd->ias_kts` 截断后再计算刻度位置。当空速从 250.1→250.9 变化时（8 帧 × 16ms = 128ms），刻度纹丝不动，产生视觉"粘滞感"。

**修复方案**：改为 `float ias = fd->ias_kts`，用**当前浮点值**而非截断整数计算每个刻度线的像素偏移：
```c
// 每个刻度位置 = (tick_value - current_float) * pixels_per_unit
float tick_y = ref_y - (tick_value - ias) * px_per_kt;
```

**为什么这很重要**：答辩老师如果问"你写 PFD 跟别人有什么不同？"——这就是答案。大多数学生项目用整数截断，我们实现了**机械仪表级的平滑度**。

#### 答辩关键词
> "PFD 的核心技术难点是**浮点连续滚动**——我们用 `float` 差值而非 `int` 截断来计算刻度位置，消除了 128ms 级别的视觉粘滞，达到了工业级飞行模拟器的平滑度标准。"

### 🏆 创新点 8：FMC CDU 手动航路点编辑

#### 概念解释

**真实 B737 FMC 的航路编辑流程**：
1. 在 CDU 便签本（Scratchpad）输入航路点标识符
2. 按 LEGS 页面对应行的**行选键（LSK）**
3. 左 LSK = 插入到该行**之前**，右 LSK = 插入到**之后**
4. 清空便签本按右 LSK = **删除**该行

#### 我们的设计思维

**三层架构**：
```
CDU 界面层 (fmc.c)
  ├─ handle_lsk() → 解析 LSK 操作意图
  ├─ find_waypoint_by_ident() → HashTable O(1) 查找
  └─ flight_plan_add_waypoint() / _remove_waypoint()
      └─ 数据层 (navdata.c)
           └─ memmove() 移位 + flight_plan_recalculate()
```

**为什么用 HashTable 而不是线性搜索？**
- 23.5 万条目的线性搜索在用户每次点击 LSK 时都会触发
- O(n) strcmp × 235,000 ≈ 明显的 UI 延迟
- 我们在 FMC 初始化时构建 50 万桶哈希表（~0.47 负载因子），插入后 `ht_get()` 是 O(1)

#### 答辩关键词
> "我们把 FMC 从'自动规划黑盒'升级为**完全手动可控的航路编辑器**——飞行员可以像真实驾驶舱一样逐点构建航路，这是飞行训练价值的核心体现。"

### Q1：为什么选择 C 语言而不是 Python/C++/C#？

**预期频率**：⭐⭐⭐⭐⭐（几乎必问）

**回答要点**：
1. **实训目标匹配**：课程要求掌握 C 语言在系统级开发中的高级编程技巧——指针、内存管理、结构体设计。C 语言是最能体现这些能力的语言。
2. **性能需求**：8 块仪表 × 60 FPS 实时渲染 + UDP 网络收发 + 音频合成 + AI WebSocket，需要**确定性低延迟**。C 语言没有 GC 暂停，没有虚函数开销。
3. **数据结构从零实现**：用 C 手写哈希表/AVL 树/空间哈希/链式前向星，相比于 C++ STL 或 Python dict，能**展示对底层原理的真正理解**。
4. **跨平台对接**：X-Plane 的 UDP 协议是二进制格式（`float32_le` + null-padded strings），C 语言的 `struct` + `memcpy` 天然适合二进制协议解析。

**不要说的**：
- ❌ "因为课程要求用 C"（太被动）
- ❌ "因为 C 是最快的语言"（太笼统，且 C++ 也可以很快）

### Q2：为什么不直接使用现成的数据结构库（STL/GLib）？

**预期频率**：⭐⭐⭐⭐

**回答要点**：
1. **教学意义**：实训的知识目标明确要求"了解哈希表、链表、平衡二叉树等核心数据结构的设计原理与 C 语言实现方法"。自己实现才能**真正理解**红黑树旋转、哈希冲突解决等底层机制。
2. **定制化需求**：现成库无法满足我们的专用场景。例如：
   - 空间哈希需要**航向优先的邻域搜索**，通用 GIS 库做不了
   - 航路图的链式前向星需要**高度层约束过滤**，通用图库不支持
3. **零依赖原则**：项目目标是单个可执行文件即可运行。引入 GLib 等库会增加编译依赖和 DLL 分发复杂度。
4. **代码量证明**：6 种数据结构共 1,400+ 行，每一个都是独立设计、测试、集成的——这是**工作量和技术深度的直接证据**。

### Q3：A* 如何保证找到的是最短路径？启发函数会不会导致错误？

**预期频率**：⭐⭐⭐⭐

**回答要点**：
1. **可纳性证明**：我们的启发函数 `h(n)` = Haversine 大圆距离（地球表面 n→终点的最短弧长）。因为**直线（大圆）永远 ≤ 任何曲线路径**，所以 `h(n) ≤ h*(n)`（真实最短距离），A\* 数学上保证最优解。
2. **一致性（单调性）**：大圆距离满足三角不等式 `h(n) ≤ d(n, n') + h(n')`，因此 A\* 不会重复展开已关闭节点。
3. **实验验证**：在已知航路（如北京→上海 A593 航路）上，A\* 的输出与实际航路一致。

**追问"如果航路数据不完整怎么办？"**：
- 我们有 200 NM fallback 机制：如果两个机场间没有航路连接，自动建立直连边，保证始终能找到路径。

### Q6：空间哈希和传统 KD-Tree/R-Tree 有什么区别？为什么选空间哈希？

**预期频率**：⭐⭐⭐

**回答要点**：
1. **实现复杂度**：R-Tree 的插入需要节点分裂（split），KD-Tree 需要递归构建。空间哈希的插入只需要 `grid_key = (lat_band << 32) | lon_sector`，一行代码。
2. **C 语言友好性**：空间哈希 = 哈希表 + 链表，全部用 C 的 `calloc` + `next` 指针实现。R-Tree 需要处理节点 MBR 的更新，在 C 中实现复杂且容易出错。
3. **查询模式的特殊性**：飞机查询是**移动窗口查询**（每次查询位置接近上次），不是随机点查询。空间哈希的网格边界是固定的，可以利用空间局部性。
4. **航向优先优化**：这是我们的**独创优化**——按 cosine 相似度排序邻域网格，让航线前方的点最先返回。R-Tree 做不了这种语义排序。

### Q7：如何保证多线程安全？会不会出现数据竞争？

**预期频率**：⭐⭐⭐⭐

**回答要点**：
1. **核心模式：Mutex Snapshot**：
   ```c
   // UDP 线程（生产者）
   flight_data_update(fd, &new_data);  // lock → memcpy → unlock
   
   // 渲染线程（消费者）
   flight_data_snapshot(fd, &snap);    // lock → memcpy → unlock
   ```
   临界区只有一次 `memcpy`（~2KB），极短（<1μs），两个线程几乎不会互相阻塞。
   
2. **锁粒度**：每个子系统有独立的 mutex——`FlightData.mutex`（飞行数据）、`FMCState.mutex`（飞行计划）、`MapDisplay.data_mutex`（客舱地图）。不存在全局大锁。
3. **音频线程**：SDL2 音频回调独立线程，使用原子操作写入 tone 状态，与主线程零冲突。

**追问"有没有遇到过死锁？"**：
- 目前没有。因为我们遵循"单向依赖"原则——UDP 线程只写 FlightData 和 FMCState，渲染线程只读。不存在 A 锁了等 B、B 锁了等 A 的循环等待。

### Q8：X-Plane 11 和 X-Plane 12 的 UDP 协议有什么关键差异？

**预期频率**：⭐⭐⭐

**回答要点**：
1. **包格式差异**：
   - DATA 包：XP11 用 `"DATA\0"` (第 5 字节 = 0x00)，XP12 用 `"DATA*"` (0x2A)
   - DREF 包：XP11 可变长 42-44B，XP12 **固定 509B**（float 4B + path 400B null-padded + 100B padding）
2. **数据组编号全部重排**：N1 从 group 26→41，EGT 从 34→47，AP 从 63→108+118……
3. **多组合并**：XP12 把所有数据组合并到一个 UDP 包发送（6233 字节），XP11 分包发送
4. **单位制**：XP12 全部英制（ft, ft/min, lbs/hr），XP11 混合使用公制/英制

**这是展示"协议逆向工程能力"的好素材——准备一个对比表格打印出来。**

### Q9：你们项目最大的技术难点是什么？怎么解决的？

**预期频率**：⭐⭐⭐⭐⭐

**推荐回答（选 1-2 个讲）**：

**选项 A：XP12 协议调试**
- 没有官方文档说明 XP11→XP12 的协议变化
- 通过 Wireshark 抓包 + 逐字节对比，花了 7 个阶段逐步解决
- 体现了**底层调试能力和耐心**

**选项 B：PFD 姿态仪天地染色**
- 旧代码的地平线位置用 `(float)(0) * sin_r`（恒为 0），根本不存在
- 重写为基于地平线两端点的**逐行交叉积**判定算法
- 同时发现并修复了颜色宏在三元运算符中的逗号截断 bug
- 体现了**从 bug 表象追到根因的系统调试思维**

**选项 C：23.5 万条目的空间哈希性能**
- 初始散列函数导致某些网格冲突严重（2000+ 条目挤在同一链）
- 调优：DJB2 哈希 + 2003 素数桶数 + 1° 纬度带粒度选择
- 通过二分搜索确定最优网格大小（太大→单格条目过多，太小→邻域搜索格数过多）
