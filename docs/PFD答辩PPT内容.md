## 一、有没有技术创新？

### 1. 全 float 亚像素连续滚动

所有刻度带用 `float` 计算像素位置，非 `int` 截断。`int` 截断时 250.1→250.9kt 的 8 帧（~128ms）刻度丝纹不动，肉眼可见"粘滞感"。`float` 逐像素滑动。

> 💡 **"px_per_kt=5.0 怎么定出来的？窗口缩放还能对齐吗？"**

**答**：`px_per_kt` 由 `(空速带高度) / (可见速度范围)` 计算得出，不是硬编码常量。当前设计可见范围 ±80kt，带宽高约 400px，所以 400/80=5.0。窗口缩放到 800×600 时，布局按**比例重算**（`PFDLayout` 每帧 `compute_layout()`），`px_per_kt` 自动适配新尺寸，刻度始终对齐。

### 2. 交叉积逐行天地染色

姿态仪地平线不是简单的"上半蓝下半棕"——滚转时地平线倾斜，每行独立用 2D 向量交叉积判定该像素在天上还是地面。

**交叉积原理**：给定有向线段 A→B 和任意点 P，2D 交叉积 `(B-A) × (P-A)` 的符号判定 P 在线的左侧还是右侧：

```
cross = ldx × (Py - Ay) - ldy × (Px - Ax)
     > 0  →  P 在线左侧
     < 0  →  P 在线右侧
     = 0  →  P 在线上
```

代码中 `ldx = rr.x - lr.x`、`ldy = rr.y - lr.y` 是旋转后地平线的方向向量。`cross < 0` 为天空（蓝），`cross > 0` 为地面（棕）。对每条扫描线的左右端点各算一次交叉积——同号则整行单色，异号则地平线穿过该行，解线性方程求交点坐标后左右分色。

**退化问题**：当 `ldy ≈ 0`（地平线接近水平，即滚转角 ≈ 0° 或 ±180°），求交点的公式 `x = lr.x + ldx·dy / ldy` 出现除以零。但此时地平线是水平的——**每行的左右端点交叉积必然同号**，整行单色渲染，代码根本不会进入分支求交点的逻辑。退化被算法结构自然规避，`fabsf(ldy) > 0.0001f` 是双重保险。

> 💡 **"如果滚转 90°，这套算法还能工作吗？"**

**答**：能。滚转 90° 时 `ldy` 最大（地平线完全垂直），交叉积判定不受影响，且分支求交点时 `ldy` 远离零，精度最优。真正可能出问题的是滚转 0°（地平线水平），但如上所述已被结构规避。

## 二、实现了哪些基本功能？

| 区域 | 功能 |
|------|------|
| **FMA**（顶） | 4 列 × 3 色：A/THR / ROLL / PITCH / AP，白色=预位，绿色=活跃，品红=管理模式 |
| **N1 推力** | 双发水平条形图，绿→琥珀→红渐变，红色 100% 参考线 |
| **姿态仪**（中） | 天地染色 + 俯仰阶梯（每 2.5°/10°）+ 滚转指针，可见 ±35° 俯仰 |
| **空速带**（左） | `px_per_kt=5.0`，每 10/20kt 刻度，失速红区 <110kt，超速红区 >340kt |
| **高度带**（右） | `px_per_ft=0.6`，每 100ft 刻度，BARO 气压窗 |
| **VSI**（右） | ±6,000fpm 非线性两段式刻度，青色三角指针 |
| **航向带**（底） | 每 5°/10° 刻度，`exp_smooth_angle()` 处理 359°→0° 环绕，AP 预选 Bug |
| **平滑系统** | 8 参数独立指数平滑（α=0.10~0.20），消除 UDP 20Hz 数据的帧间跳变 |
