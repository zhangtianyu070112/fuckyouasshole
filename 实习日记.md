# 实习日记 — 航空驾驶舱模拟控制系统

---

## 第一天 · 环境搭建与项目背景

今天算是正式开工。上午主要在折腾环境——在 Windows 上搭 MSYS2 + GCC 编译链，网络不行，默认镜像源全 404，换了清华源才把 pacman 跑通。装了 mingw-w64-x86_64-SDL2 全家桶（SDL2、SDL2_image、SDL2_ttf、SDL2_gfx），然后用 VSCode 配好了 C/C++ 插件和 tasks.json，试了一下 hello world 级别的小程序，编译链接都正常。

下午通读了实训说明文档，对整体目标有了概念：用纯 C + SDL2 做一个商用客机驾驶舱的仪表仿真系统，覆盖 PFD、ND、EICAS 三大仪表盘，还要集成 FMC 飞行管理计算机和 X-Plane 11 的外部数据对接。说实话刚看到这个项目量级的时候有点懵——以前只写过控制台程序，图形编程完全是零基础。不过仔细看了一下 SDL2 的示例代码，渲染管线其实挺清晰的：初始化窗口 → 创建渲染器 → 绘制 → 呈现，和之前接触过的 Canvas 思路有点像。明天先拿 SDL2 做几个小实验熟悉 API，稳扎稳打。

**反思**：环境配置花了比预期多的时间，主要是镜像源的问题没提前预判。之后遇到基础设施类的问题应该先搜一圈常见坑再动手，不然反复踩坑太磨人了。

---

## 第二天 · SDL2 绘图实验与纹理理解

今天的主要任务是摸清 SDL2 的绘图流程。上午写了几个测试程序：画矩形、画线条、画填充圆、加载 PNG 图片。一开始对渲染目标的概念不清楚，直接在 window surface 上画，结果刷新的时候疯狂闪屏。查了文档才搞明白——SDL2 的正确姿势是用 `SDL_Renderer` 做后端渲染，先 `SDL_RenderClear` 清屏，再调各种 `SDL_RenderDraw*` / `SDL_RenderCopy*`，最后 `SDL_RenderPresent` 一次性呈现，双缓冲机制自己就解决了撕裂问题。

下午重点研究纹理的用法。仪表盘上那些刻度线、数字、指针，如果每一帧都重新用几何图形画一遍，性能肯定扛不住。所以思路是：静态内容（刻度背景、固定标签）预渲染成 `SDL_Texture`，存起来，每帧只需要 `SDL_RenderCopy` 贴上去就行；动态部分（指针、数值）再实时绘制。这个过程有点像游戏开发里的 sprite sheet 思路，把"画"和"贴"分开，能省大量 CPU。另外还踩了一个坑：`SDL_ttf` 渲染文字返回的是 surface，需要 `SDL_CreateTextureFromSurface` 转成纹理才能用 Renderer 画，这个转换一开始漏了，文字怎么也出不来，debug 了快一个小时。

**反思**：今天最大的收获不是学会了哪几个 API，而是理解了"预渲染→纹理缓存→每帧贴图"这个管线。做实时图形程序，性能意识要从第一天就建立。遇到闪屏问题的时候第一反应不该是瞎调参数，而是先理解渲染模型。

## 第三天 · PFD 主飞行显示器窗口绘制

今天开始啃硬骨头——PFD（主飞行显示器）。PFD 是驾驶舱里最核心的仪表，集成了空速、姿态、高度、航向、垂直速度五大信息，布局复杂，分区多。参考了 Boeing 737 的真实 PFD 照片，先规划了布局比例：顶部 FMA 飞行模式通告 5%，中间主体约 70%（空速带左侧 10%、姿态仪居中 58%、高度带右侧 10%、VSI 垂直速度右侧 7%），底部航向带 8%。

上午用 SDL2 的 `SDL_RenderDrawRect` 和 `SDL_RenderFillRect` 把框架搭出来了，分段画了几个主要区域的分隔线。姿态仪的天地区域最难画——上半部蓝色（天空）、下半部棕色（地面），中间一条白色的人工地平线。一开始用纯色矩形填充，效果很生硬，后来加了渐变色过渡，视觉效果好了不少。空速带和高度带的刻度线用循环批量绘制，每个刻度线标注数值，再用 `draw_text_simple` 把 TTF 字体渲染上去。

下午遇到了一个坑：TTF 字体加载后中文显示全是方框。排查发现 B612 字体是纯英文的，没有 CJK 字符集。好在仪表盘标注基本都是英文缩写（SPD、ALT、HDG、V/S），暂时不影响。但后面做 ND 导航显示的时候需要考虑中文字体方案。

**反思**：画 UI 之前花半小时画布局草稿是值得的，确定好比例之后再动手写代码，调整量小很多。直接上手写容易画着画着发现位置全歪了，返工更耗时。

## 第四天 · 动态数据驱动 PFD

今天让昨天画好的 PFD"活"起来了。核心思路是建一个 `FlightData` 结构体，存放空速、高度、航向、垂直速度、姿态角等字段，然后在渲染循环里根据这些值动态更新仪表显示。

空速带和高度带用 `float` 做连续滚动：当前值不是整数的时候，刻度条的位置要逐像素平滑移动。一开始用 `int` 截断计算刻度位置，结果数值变化时刻度纹丝不动，肉眼看明显卡顿。改成 `(tick_value - current_float) * px_per_unit` 的差值计算之后，滚动丝滑了。航向带同理，从固定偏移量改成了 `(h - hdg) * px_per_deg` 的连续差值算法。

姿态仪的俯仰和滚转是今天最花时间的部分。俯仰角驱动地平线上下平移，滚转角驱动整个姿态区域旋转。旋转用 `SDL_RenderCopyEx` 传 angle 参数实现，但一开始旋转中心算错了——`SDL_RenderCopyEx` 的 center 参数是相对于纹理的，不是屏幕坐标，偏移量得仔细算。改了几版参数才对齐。用模拟数据跑了一下从起飞滑跑到爬升的全过程，PFD 上五大数据同步变化，终于有点驾驶舱的感觉了。

**反思**：动态指针和数据结构的设计比渲染本身更重要。`FlightData` 结构体定义得好，各个仪表模块各自读取需要的字段，模块间耦合低，后面加新仪表不需要改渲染管线。另外浮点平滑计算不是花活，是实时仪表的基本功——`int` 截断在 60fps 下会非常明显，以后做这类东西默认上浮点。
